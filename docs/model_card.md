# Model Card — Fraud Detection v1

Based on the [Model Cards for Model Reporting](https://arxiv.org/abs/1810.03993) template.

## Model details

| Field               | Value                                                                  |
|---------------------|------------------------------------------------------------------------|
| Model name          | `fraud-detection-lgbm-v1`                                              |
| Algorithm           | LightGBM (gradient boosted trees) with isotonic probability calibration|
| Input               | One transaction (Pydantic `Transaction` schema, ~20 features)          |
| Output              | Probability of fraud in `[0, 1]`                                       |
| Library versions    | LightGBM >= 4.3, scikit-learn >= 1.4                                   |
| Owner               | Risk DS team                                                           |
| Contact             | `mlops@fintech.example`                                                |
| Versioning          | `model_version = YYYYMMDD-HHMMSS` set at training time                 |

## Intended use

* **Primary use case** — Real-time scoring of credit-card transactions in
  the authorisation flow. The score, combined with business rules, drives
  the APPROVE / REVIEW / DECLINE decision.
* **Out of scope** — Not designed for fraud investigation post-hoc, not
  designed for credit risk assessment, not designed to be used as the sole
  input of any automated decision affecting a customer (a HARD_BLOCK rule
  must also fire for a DECLINE in critical flows).

## Training data

* **Source** — Synthetic data generated by `data.synthetic`. The generator
  implements five canonical attack patterns (card testing, geo anomaly,
  cashout, account takeover, high-risk MCC) plus a customer-level baseline
  for legitimate transactions.
* **Volume** — 200 000 transactions per training run, fraud rate ≈ 1.2%.
* **Split** — Time-based: oldest 80% → training, newest 20% → test. Inside
  the training set, the last 15% (by timestamp) is held out as the
  validation set used for early stopping and probability calibration.
* **PII** — None. All identifiers are synthetic.

In production, the synthetic generator is replaced with a feature-store
extraction job; the schema and feature columns remain identical.

## Features

See `src/fraud_detection/features/build_features.py` for the canonical
list. Grouped here:

* **Amount-based** — `amount`, `amount_log`, `amount_ratio_avg_30d`, `amount_zscore_30d`
* **Velocity** — `n_tx_last_1h`, `n_tx_last_24h`, `velocity_score`, `is_burst_1h`
* **Temporal** — `hour`, `is_night`, `is_weekend`
* **Geo** — `is_geo_mismatch`
* **Channel / risk** — `is_cnp`, `is_ecom`, `is_atm`, `is_high_risk_mcc`
* **Customer** — `customer_age_days`, `card_age_days`, `amount_avg_30d`,
  `amount_std_30d`, `distinct_countries_last_24h`

## Evaluation

Reported on the held-out time-based test set (run via `make train`).

| Metric                      | Why we track it                                                      |
|-----------------------------|----------------------------------------------------------------------|
| ROC AUC                     | Overall ranking quality                                              |
| **PR AUC**                  | Robust to class imbalance — primary headline metric                  |
| **Recall @ 1% FPR**         | Operational: how much fraud do we catch while keeping false alarms low |
| Precision @ top 1%          | Manual review workload: of our top-1% riskiest, how many are fraud   |
| F1 @ 0.5                    | Reference only; thresholds in prod come from `model.yaml`            |

## Fairness & bias

The current model does not consume any sensitive attributes (no
nationality, no age band, no gender). Geographic features (country) are
present and motivated by anti-fraud regulation, but we monitor decline
rates per `card_country` in production and alert on outliers.

## Limitations

1. **Concept drift** — Fraudsters adapt. Retrain monthly; alert on PSI > 0.25
   on any top-5 feature.
2. **Data quality of velocity features** — `n_tx_last_1h`, `n_tx_last_24h`,
   `distinct_countries_last_24h` come from the feature store. Stale data
   in the feature store will silently degrade the model. Monitor freshness.
3. **Highly imbalanced labels** — fraud labels are subject to a ground-truth
   delay (chargebacks confirmed weeks later). The training set always
   slightly under-counts fraud.

## Decision policy

The decision policy is part of the model contract:

| Adjusted probability | Action  |
|----------------------|---------|
| < 0.30               | APPROVE |
| 0.30 to 0.70         | REVIEW  |
| ≥ 0.70               | DECLINE |

A HARD_BLOCK rule forces DECLINE regardless of the score.
SUSPECT rules each add a small uplift to the score (capped at +0.20).

Thresholds are configured in `configs/model.yaml` and reviewed monthly
against business KPIs (false decline rate, manual review capacity).

## Ethical considerations

A false decline at point-of-sale is a poor customer experience and a
real financial impact (lost sale). The threshold policy is set in
collaboration with the Customer Care team to keep false decline rate
within agreed SLAs.

A false approval costs the business directly (chargeback). The DS team
monitors recall closely and alerts on any drop > 5% week-over-week.
